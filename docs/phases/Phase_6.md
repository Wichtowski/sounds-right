# Phase 6 — Publishing

## Purpose

Implement the publishing layer for **Sounds Right**.

By the end of Phase 6, approved transcript versions should be publishable as stable public karaoke artifacts. The system should expose public read endpoints for published transcripts while keeping the review/transcription internals private.

Phase 6 connects the result of the review flow to a production-facing output.

The main goal is:

```txt
Approved track version -> published artifact -> stable public API endpoint
```

This phase should not add new transcription logic. It should focus on safe publishing, public artifact layout, stable URLs, manifest generation, and published-state management.

---

## Final Phase 6 Goal

At the end of Phase 6, the app should support this flow:

```txt
1. A track version has completed transcription.
2. A reviewer approves the version.
3. An admin/reviewer publishes the approved version.
4. The system writes stable public transcript artifacts to MinIO.
5. The system marks the version as published.
6. The public API returns the published karaoke manifest/transcript.
7. The frontend can open a public preview page for the published track.
```

Public users should be able to fetch published transcript data without authentication.

Internal draft/review/transcription artifacts must remain private.

## Key Product Rule

Only an approved version can be published.

Valid high-level lifecycle:

```txt
draft
  -> upload_url_created
  -> uploaded
  -> queued
  -> processing
  -> completed
  -> pending_review
  -> approved
  -> published
```

Rejected versions cannot be published.

Failed versions cannot be published.

Draft/uploaded/processing versions cannot be published.

---

## Publishing Design

The system should separate **review artifacts** from **published artifacts**.

Review/internal artifacts are generated by the transcription worker and are used by reviewers.

Published artifacts are stable, clean, public-facing files intended for the public API and future frontend/player usage.

Recommended buckets:

```txt
sounds-right-transcripts      internal/review transcript artifacts
sounds-right-artifacts        internal derived artifacts, previews, waveform files
sounds-right-public           public/published karaoke artifacts
```

If `sounds-right-public` does not exist yet, add it.

Alternative:

Use one bucket with separate prefixes:

```txt
internal/transcripts/...
public/karaoke/...
```

Recommended for clarity:

```txt
Use a separate public bucket: sounds-right-public
```

## Public Object Layout

Published files should use stable object keys.

Recommended layout:

```txt
karaoke/
  {artist_slug}/
    {track_slug}/
      manifest.json
      latest.json
      versions/
        v{version}/
          manifest.json
          transcript.json
          segments.json
          words.json
```

Example:

```txt
karaoke/kendrick-lamar/squabble-up/manifest.json
karaoke/kendrick-lamar/squabble-up/latest.json
karaoke/kendrick-lamar/squabble-up/versions/v1/manifest.json
karaoke/kendrick-lamar/squabble-up/versions/v1/transcript.json
karaoke/kendrick-lamar/squabble-up/versions/v1/segments.json
karaoke/kendrick-lamar/squabble-up/versions/v1/words.json
```

`latest.json` should point to the current published version or contain the full latest transcript depending on the implementation.

Recommended:

```txt
manifest.json -> metadata about track and available published versions
latest.json   -> full latest published transcript
versions/vN   -> immutable versioned files
```

---

## Important Immutability Rule

Versioned published artifacts should be immutable.

Do not overwrite:

```txt
karaoke/{artist}/{track}/versions/v1/transcript.json
```

If a newer approved transcript is published, create:

```txt
karaoke/{artist}/{track}/versions/v2/transcript.json
```

Then update mutable pointers:

```txt
karaoke/{artist}/{track}/manifest.json
karaoke/{artist}/{track}/latest.json
```

This gives stable history and easy rollback.

## Database Changes

Add a `publications` table.

### publications

```txt
publications
- id uuid primary key
- track_id uuid not null references tracks(id)
- track_version_id uuid not null references track_versions(id)
- version integer not null
- status text not null
- public_manifest_object_key text not null
- public_latest_object_key text nullable
- public_transcript_object_key text not null
- public_segments_object_key text nullable
- public_words_object_key text nullable
- published_by_user_id uuid nullable references users(id)
- published_at timestamptz nullable
- unpublished_at timestamptz nullable
- created_at timestamptz not null
- updated_at timestamptz not null
```

Publication statuses:

```txt
published
unpublished
superseded
```

Recommended constraints:

```txt
unique(track_version_id)
unique(track_id, version)
```

Optional but useful:

```txt
Only one active publication per track where status = 'published'
```

This can be enforced in application logic first. A partial unique index can be added later if desired.

## Track Version Status Updates

Ensure `track_versions.status` supports:

```txt
approved
published
```

When publishing succeeds:

```txt
track_versions.status = published
```

If an older version was already published, mark it as superseded in `publications`, but do not delete its versioned public files.

Possible behavior:

```txt
v1 published
v2 approved
publish v2
=> v1 publication status = superseded
=> v2 publication status = published
=> track latest points to v2
```

Do not mutate old transcript objects.

### POST /api/versions/{version_id}/publish

Requires auth.

Allowed roles:

```txt
reviewer
admin
```

If role enforcement is still simple, at minimum require authenticated user and document stricter RBAC for later.

Request:

```json
{
  "make_latest": true
}
```

Default:

```txt
make_latest = true
```

Behavior:

```txt
1. Load track version.
2. Verify version status is approved.
3. Verify transcript artifact exists in internal MinIO storage.
4. Load transcript JSON from internal object storage.
5. Validate transcript schema.
6. Generate public manifest.
7. Write immutable versioned public files to MinIO.
8. Update latest.json and manifest.json if make_latest is true.
9. Create publication row.
10. Mark previous active publication as superseded if needed.
11. Mark track version as published.
12. Emit publication.published event if event system is ready.
13. Return publication details.
```

Response:

```json
{
  "id": "uuid",
  "track_id": "uuid",
  "track_version_id": "uuid",
  "version": 1,
  "status": "published",
  "public_urls": {
    "manifest": "/api/public/karaoke/kendrick-lamar/squabble-up/manifest",
    "latest": "/api/public/karaoke/kendrick-lamar/squabble-up/latest",
    "version": "/api/public/karaoke/kendrick-lamar/squabble-up/versions/1"
  },
  "published_at": "2026-07-04T12:00:00Z"
}
```

Errors:

```txt
401 unauthorized
403 forbidden
404 version not found
409 version is not approved
409 version already published
500 publish failed
```

Do not partially mark DB state as published if object writes failed.

Use a transaction for DB updates after successful object writes.

---

### POST /api/versions/{version_id}/unpublish

Optional but recommended.

Requires auth.

Allowed roles:

```txt
admin
```

Request:

```json
{
  "reason": "Bad timestamps in chorus"
}
```

Behavior:

```txt
1. Load active publication for version.
2. Mark publication as unpublished.
3. Optionally remove latest pointer if this was latest.
4. Do not delete immutable versioned files by default.
5. Emit publication.unpublished event if event system is ready.
```

Important:

Unpublish should not hard-delete artifacts by default.

Deletion can be a separate admin maintenance action later.

## Transcript Schema Validation

Before publishing, validate the transcript artifact.

Minimum validation:

```txt
schema_version exists
track exists
metadata exists
segments exists and is a list
segments have start/end/text/words
words have word/start/end
```

If validation fails:

```txt
Do not publish.
Return 409 or 422 with safe error.
```

Do not return raw stack traces.

```txt
1. Read internal transcript object.
2. Optionally split transcript into transcript/segments/words files.
3. Write versioned public objects.
4. Write latest.json.
```

Recommended content types:

```txt

```

Recommended metadata:


Calculate SHA-256 for published transcript content.

## Transaction Strategy

Publishing touches both object storage and Postgres.

There is no true transaction across MinIO and Postgres.

Recommended pragmatic sequence:

```txt
1. Validate DB state.
2. Read internal transcript object.
3. Prepare public JSON files in memory/temp.
4. Upload public versioned objects to MinIO.
5. Upload latest/manifest objects to MinIO.
6. Open DB transaction.
7. Mark previous publication superseded if needed.
8. Insert publication row.
9. Update track_version status to published.
10. Commit DB transaction.
11. Emit event or enqueue outbox record.
```

If DB commit fails after object upload:

```txt
Log error.
Return failure.
Leave orphaned public objects for cleanup job later.
```

Optional cleanup:

```txt
Delete newly uploaded public objects if DB commit fails.
```

Do not mark DB as published before object writes succeed.

## Acceptance Criteria

Phase 6 is complete when:

```txt
1. Approved track version can be published.
2. Non-approved versions cannot be published.
3. Publication creates a database row.
4. Publication writes stable public objects to MinIO.
5. Public manifest is generated.
6. Public latest transcript is generated.
7. Public versioned transcript is immutable.
8. Publishing v2 supersedes v1 but keeps v1 available by version URL.
9. Track version status changes to published.
10. Public API exposes manifest/latest/version endpoints.
11. Public endpoints require no auth.
12. Internal publish endpoint requires reviewer/admin auth.
13. Frontend can trigger publishing from an approved version page.
14. Frontend can display public published karaoke page.
15. Envoy routes public API endpoints correctly.
16. Tests cover successful and invalid publishing cases.
17. No new transcription logic is added.
18. Raw MinIO internal object URLs are not the official public API.
```
